## 便宜 VPS 能跑，但前提是系统必须“朴素”

$5 的 VPS 容错空间非常小。想跑稳，就要把部署做得足够朴素：一个进程管理器、一个反向代理、受控日志、尽量少的后台任务。

VPS 的价值是成本可控、行为可预测。你放弃了平台自动扩缩容，所以整套东西必须在凌晨两点也能靠 SSH 看懂、靠几条命令恢复。

## 什么时候应该用它

- 你需要 Hermes 7x24 在线，但不想承担无服务器按量突增成本。
- 工作负载稳定且不大，一台小 Linux 机器就能扛住。
- 你更喜欢直接 SSH + systemd，而不是托管平台抽象。

## 参考工作流

1. 准备一台最小化 Linux 主机，并创建非 root 应用用户。
2. 安装 Node，拉代码，构建生产包。
3. 通过 systemd 和反向代理把 Hermes 跑起来。
4. 补上备份、日志轮转、健康检查和基础告警。

## 步骤 1: 在复制代码前先把主机收拾干净

先创建独立用户、开防火墙，并决定应用代码、日志和环境变量文件放哪。主机目录结构如果一开始混乱，后面运维一定很痛。

```bash
sudo adduser hermes
sudo ufw allow OpenSSH
sudo ufw allow 80
sudo ufw allow 443
sudo ufw enable
```

## 步骤 2: 用 systemd 做“无聊但可靠”的重启

小 VPS 不需要复杂编排。一个带失败自动重启、明确工作目录和环境文件支持的 systemd unit，通常就够了。

## 步骤 3: 把 DNS 和 HTTPS 视为部署的一部分

只在 localhost 能打开，不叫部署完成。上线前必须确认 DNS 解析、证书签发和外网可达性。

## 上线前检查清单

- 服务必须跑在非 root 用户下。
- 打开自动重启和持久化日志。
- 从机器外部验证 DNS、HTTPS 和健康检查。
- 对环境变量文件和不可重建的本地状态做备份。

## 常见问题

### $5 的 VPS 真能稳定跑 Hermes 吗？

对于轻量、平稳的负载可以。一旦任务变得突发、吃内存或对时延敏感，瓶颈会很快出现。

### 要不要上 PM2？

很多部署场景下 systemd 就够了，而且减少一个移动部件。PM2 也可以用，但不是必须。

### 便宜 VPS 最先出问题的通常是什么？

很多时候不是 CPU，而是磁盘和内存压力，尤其在日志无限增长或构建也在同一台机器上进行时。
