## 为什么技能质量决定上限

做一个演示能跑通的技能并不难，难的是模型变了、权限变了、接手的人变了以后，这个技能还能稳定工作。这篇文章讲的是工程化写法，不是“先跑起来再说”的写法。

- 先设计契约，再写实现
- 失败路径要显式
- 副作用要尽量小，而且要可观测
- 既要有单测，也要有 CLI 冒烟测试

## 先把契约写清楚

在写逻辑前，先定义技能接收什么、返回什么、允许操作什么。最理想的状态是：不看实现，也能知道技能怎么用。

```typescript
export interface DeployParams {
  service: string;
  environment: 'staging' | 'production';
  confirm?: boolean;
}

export interface DeployResult {
  success: boolean;
  summary: string;
  deploymentId?: string;
  warnings?: string[];
}
```

如果一个技能的入参与出参很难用小接口描述清楚，通常说明它做的事太多了，应该拆成规划技能和执行技能。

## 输入校验要发生在副作用之前

不要把不完整输入直接传给数据库、Shell 或外部 API。校验要在任何副作用发生前完成。

```typescript
function assertDeployParams(params: DeployParams) {
  if (!params.service) throw new Error('service is required');
  if (!['staging', 'production'].includes(params.environment)) {
    throw new Error('environment must be staging or production');
  }
}
```

## 测试至少分三层

### 第一层：单元测试

把文件系统、HTTP、Hermes context 都 mock 掉，专门验证分支逻辑和错误处理。

```bash
npm test -- skill-development-best-practices
```

### 第二层：CLI 冒烟测试

通过真实 Hermes CLI 跑一遍，检查参数序列化和结果展示。

```bash
hermes skill test deploy-review --params '{"service":"billing","environment":"staging"}'
```

### 第三层：失败演练

刻意模拟缺凭证、超时、空输入、部分字段缺失等情况。线上事故大多出在边界，而不是 happy path。

## 发布前检查

- 权限是否最小化
- 配置项是否文档化
- 是否覆盖至少 1 条成功路径和 2 条失败路径
- CLI 输出是否让人看得懂
- README 是否包含安装、配置和回滚说明

## 常见问题

### 单测都过了，但在 Hermes 里失败

说明你的 mock 太理想化了。补一个 `hermes skill test` 的真实冒烟测试，看看 CLI 传参与测试夹具是否一致。

### 本地能跑，自动化里不行

通常是隐藏依赖没处理，比如交互式输入、写家目录权限、只存在于你当前 shell 的环境变量。

### 模型输出越来越飘

从自由文本切到受约束的 JSON schema，并在继续执行前先校验输出。
