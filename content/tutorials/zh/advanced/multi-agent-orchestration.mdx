## 为什么需要多 Agent？

单个 AI Agent 已经很强大。多个 Agent 协同工作则具有变革性。多 Agent 编排让你可以：

- **分解复杂任务** — 每个 Agent 专注于自己最擅长的领域
- **并行处理** — 多个 Agent 同时工作
- **质量控制** — Agent 互相审查对方的工作
- **专业分工** — 代码、写作、研究各有专属 Agent

就像一个专家团队 vs 一个通才。你不会让同一个人既当设计师、又当开发者、还当测试员。

## 核心概念

### Agent 定义

一个 Agent 是拥有特定角色、模型和技能的 Hermes 实例：

```yaml
agents:
  编码员:
    model: "anthropic/claude-3.5-sonnet"
    system_prompt: "你是一名资深软件工程师..."
    skills: ["code-runner", "github-integration"]
    
  审查员:
    model: "openai/gpt-4o"
    system_prompt: "你是一名高级代码审查员..."
    skills: ["github-integration"]
    
  写手:
    model: "openai/gpt-4o-mini"
    system_prompt: "你是一名技术文档写手..."
    skills: ["file-manager"]
```

### 编排器

编排器管理 Agent 之间的工作流：

```bash
hermes orchestrate --config workflow.yaml
```

## 模式 1：监督者模式

一个 Agent 负责分配任务和审查结果。

```
              ┌─────────────┐
              │   监督者     │
              │   (GPT-4o)   │
              └──────┬───────┘
         ┌───────────┼───────────┐
         ▼           ▼           ▼
   ┌──────────┐ ┌──────────┐ ┌──────────┐
   │  编码员   │ │  写手    │ │  测试员   │
   │ (Claude) │ │(GPT-Mini)│ │ (Claude) │
   └──────────┘ └──────────┘ └──────────┘
```

```yaml
# supervisor-workflow.yaml
orchestration:
  pattern: supervisor
  
  supervisor:
    model: "openai/gpt-4o"
    system_prompt: |
      你是项目经理。把任务拆分为子任务，
      分配给合适的 Agent，并验证结果。
    max_iterations: 10
    
  workers:
    编码员:
      model: "anthropic/claude-3.5-sonnet"
      system_prompt: "你编写生产级代码。"
      skills: ["code-runner", "file-manager"]
      
    写手:
      model: "openai/gpt-4o-mini"
      system_prompt: "你撰写清晰的文档。"
      skills: ["file-manager"]
      
    测试员:
      model: "anthropic/claude-3.5-sonnet"
      system_prompt: "你编写全面的测试。"
      skills: ["code-runner"]
```

**适用场景**：需要编码、测试和文档的完整功能开发。

### 运行示例

```bash
hermes orchestrate --config supervisor-workflow.yaml \
  "构建一个待办事项 REST API，包含 CRUD 操作、测试和文档"
```

## 模式 2：流水线模式

Agent 按顺序接力处理数据。

```
[输入] → [调研] → [初稿] → [编辑] → [排版] → [输出]
```

```yaml
orchestration:
  pattern: pipeline
  
  stages:
    - name: 调研
      model: "openai/gpt-4o"
      system_prompt: "深入调研主题。"
      skills: ["web-search"]
      
    - name: 初稿
      model: "anthropic/claude-3.5-sonnet"
      system_prompt: "撰写详细的技术文章。"
      
    - name: 编辑
      model: "openai/gpt-4o"
      system_prompt: "修改以提升清晰度、准确性和流畅度。"
      
    - name: 排版
      model: "openai/gpt-4o-mini"
      system_prompt: "格式化为带正确标题的 Markdown。"
```

**适用场景**：内容创作流水线、数据处理管道。

## 模式 3：辩论模式

两个 Agent 提出对立观点，产生平衡结论。

```
     ┌──────────┐
     │  裁判    │
     │ (GPT-4o) │
     └────┬─────┘
    ┌─────┴──────┐
    ▼            ▼
┌────────┐  ┌────────┐
│ 支持方  │  │ 反对方 │
│(Claude)│  │(Claude)│
└────────┘  └────────┘
```

```yaml
orchestration:
  pattern: debate
  rounds: 3
  
  agents:
    支持方:
      model: "anthropic/claude-3.5-sonnet"
      system_prompt: "支持该方案并给出论据。"
      
    反对方:
      model: "anthropic/claude-3.5-sonnet"
      system_prompt: "反对该方案并给出论据。"
      
  judge:
    model: "openai/gpt-4o"
    system_prompt: |
      公平评估双方论点。综合各自最佳观点
      得出平衡的结论。
```

**适用场景**：技术决策、权衡评估、探索复杂问题。

### 示例：技术选型

```bash
hermes orchestrate --config debate-workflow.yaml \
  "我们的电商平台应该用 MongoDB 还是 PostgreSQL？"
```

> **实用场景**：当你对 React vs Vue、微服务 vs 单体架构、自研 vs 买服务等决策犹豫时，辩论模式可以帮你全面分析利弊。

## 模式 4：Map-Reduce

将工作并行分发给多个 Agent，然后合并结果。

```
                    [任务]
                      │
         ┌────────────┼────────────┐
         ▼            ▼            ▼
    [Worker 1]   [Worker 2]   [Worker 3]
         │            │            │
         └────────────┼────────────┘
                      ▼
                   [汇总]
```

```yaml
orchestration:
  pattern: map-reduce
  
  mapper:
    model: "openai/gpt-4o-mini"
    system_prompt: "分析指定的代码文件。"
    parallel: true
    max_workers: 5
    
  reducer:
    model: "anthropic/claude-3.5-sonnet"
    system_prompt: |
      将所有分析结果合并为一份报告。
      关键问题排在最前面。
```

**适用场景**：多文件代码审查、批量数据分析、并行调研。

## 费用管理

多 Agent 工作流会成倍增加 API 费用。以下是管理策略：

| 策略 | 方法 |
|------|------|
| 简单 Agent 用便宜模型 | 写手/排版 → GPT-4o Mini |
| 限制迭代次数 | `max_iterations: 5` |
| 设置预算上限 | `max_cost: 0.50` (每次运行) |
| 缓存中间结果 | `cache: true` |
| 测试时用本地模型 | `model: ollama/qwen2:7b` |

```yaml
orchestration:
  cost_control:
    max_cost_per_run: 1.00
    warn_at: 0.50
    model_fallback: "openai/gpt-4o-mini"
```

### 费用预估

| 模式 | 简单任务 | 中等任务 | 复杂任务 |
|------|:--------:|:--------:|:--------:|
| 监督者（3 Agent） | ¥0.5 | ¥2 | ¥5 |
| 流水线（4 阶段） | ¥0.3 | ¥1.5 | ¥4 |
| 辩论（3 轮） | ¥0.8 | ¥3 | ¥8 |
| Map-Reduce（5 并行） | ¥1 | ¥4 | ¥10 |
